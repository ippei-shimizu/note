# じぶん Release Note フィードバック記録

次回以降の作成時に反映すべきフィードバックを記録する。

## テイスト・文体

- 冒頭文はエモくしない。淡々と事実を述べる
- 全体的に素直・簡潔・前向きなトーン
- 「倍近い」などの強調表現は避け、「多め」「超えた」程度に抑える

## Learning

- 各項目は1〜2行の要約に留める。詳細に書きすぎない
- 読書はLearningセクションの一番下に記載する

## Development

- リリースした機能は詳細に記載してよい
- 各リポジトリにGitHubリンクを付与する
- iOSアプリリリース等の大きなイベントはApp Storeリンクも含める

## Work

- 「ECサービスの開発チームに配属された」のような事業内容が推測できる記述は入れない
- 「事業部の年間ロードマップに含まれる大きめの開発タスクを任せてもらえた」→「軽めの実装から着手しはじめた」のように謙虚な表現にする
- 「チャットや朝会で実装方針について相談・提案ができるようになり、自分の提案が採用される場面もあった」→「実装方針の相談などを提案も含めて自ら行えた」のように簡潔にする
- 具体的なフィードバック内容（「6-7割で方針を相談する」等）は記載しない
- チームの雰囲気に関する記述は感謝の気持ちを込めた自然な表現にする

## KPI

- 「多角的な視点が不足」→「視点が不足」のようにシンプルに
- 社内向けの具体的アクション（「実装前の3つの問い」「プランニング対象チケットの事前確認」等）は除去
- 「他のメンバーのPRを週1〜2本読む」→「システム理解の意味も込めて、他の人のPRを読む」のように一般化
- 「業務時間外でもシステム理解の時間を取る」→「システム理解のためのコードリーディングを行う」のように具体化

## Conclusion

- 「3年後のテックリードという目標から逆算すると」のような長期目標への言及は入れない
- 「信頼される実装者」→「信頼感を少しずつ持ってもらう」のように控えめな表現にする
