## 🔥 Learning & Development

### Learning

- Cloud PraticaのAWS基礎コースを進めた。3月はコンテナオーケストレーション〜ネットワーク・ロードバランサーまでを一気に進めた。

- **ECR** : Dockerイメージの管理・タグ運用・内部構造（マニフェスト・レイヤー）を学んだ。

- **ECS** : クラスター・サービス・タスク定義の関係を整理した。Fargateでのデプロイ方式やプライベートサブネット配置の設計も理解した。

- **セキュリティグループ** : SGチェーンによる多層防御の仕組みを学んだ。

- **IAMロール** : タスク実行ロールとタスクロールの違いを理解した。

- **DNS** : 名前解決の階層構造、各種DNSレコード（A / CNAME / NS / MX / SPF等）、権威DNSとキャッシュDNSの違いを学んだ。

- **Route 53 / Certificate Manager** : ドメイン取得、マルチアカウント構成でのDNS委任設定、HTTPS証明書の取得を実施した。

- **ALB** : ルーティング・ターゲットグループ・ヘルスチェック・リスナールールの仕組みを学んだ。

## 🍀 KPI

### keep

- 入社初月で目標より多めにポイントを消化できたので、続けていく。

- レビューで受けた指摘などをしっかり復習する習慣を継続できた。

- フィードバックをもらったら即行動に移すサイクルを回せた。

- オンボーディング中にわからないことが発生した時、自分で調査しつつ「これは相談しないと解決できない」と判断してすぐ聞けた。

### Problem

- PRの品質がまだ低い。レビューでの指摘数が多く、ラリーが多かった。
  - セキュリティやSEOなどの視点が不足していた。
  - 将来の拡張性を考慮した設計ができていない場面があった。
  - おしゃれに書きすぎて可読性が落ちた。シンプルに書くべきだった。
  - テストケースの漏れやバグが残った状態でPRを出してしまった場面があった。

- スプリントプランニングで、設計や実装方針の議論に入れなかった。既存システムの理解が追いついておらず、議論の内容についていけないケースが多かった。

- サービス全体のシステム仕様・データ構造・処理フローの理解がまだ足りていない。

### Try

- システム理解の意味も込めて、他の人のPRを読む。

- 問い合わせ対応の内容を確認して、自分なりに調査してみる。

- システム理解のためのコードリーディングを行う。

### 🚀 Conclusion

- 転職して最初の1ヶ月は不安もあったが、チームの方々の丁寧なオンボーディングとフィードバックのおかげで、スムーズにチームに入れることができた。

- 一方で、PRの品質やプランニングでの議論参加など、「今の自分に足りていないこと」も明確になった。ドメイン知識の不足が根本原因であることは理解しているので、4月はタスク起点でシステム理解を広げつつ、PR品質を上げていく。

- 今は信頼感を少しずつ持ってもらうことを目標に取り組む。レビュー指摘を減らし、設計議論に少しずつ入っていくことで、チーム内での存在感を高めていきたい。
